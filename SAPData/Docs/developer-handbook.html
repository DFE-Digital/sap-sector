Architecture

Two generators:
  • GenerateRawTables.cs ‚Üí creates & loads raw tables from cleaned CSVs
  • GenerateViews.cs ‚Üí creates materialized views from raw tables using datamap.csv

GenerateRawTables.cs

This generator is responsible for the ‚Äúlanding zone‚Äù inside Postgres. It should be safe to re-run, and it should produce deterministic table names.

ResponsibilityNotes
Create raw tables Creates one table per dataset file. Columns are created from cleaned CSV headers. A hash-based suffix is used to avoid name-length collisions.
Bulk load Loads data from *.clean.csv (or equivalent cleaned output). Prefer Postgres COPY semantics for performance.
Idempotency Should support drop-and-recreate (or truncate-and-reload) to keep raw tables consistent. 
Example outputs:
- raw_202425_performance_3979aec1
- raw_edubasealldata_20251022_a1b2c3d4

Cleaning step (before raw loading)

The cleaning step ensures:

  • headers become valid Postgres identifiers (or consistently quoted)
  • broken rows / malformed quoting are fixed
  • encoding is normalised (e.g. UTF-8)
  • numeric columns can be safely parsed by downstream clean_int / clean_numeric
If your ingestion pipeline currently uses *.clean.cs to produce cleaned output, GenerateRawTables should treat that cleaned output as the source of truth for column names.

GenerateViews.cs

GenerateViews creates one materialized view per model (e.g. establishment performance, LA performance, etc.). It uses CASE-WHEN aggregation inside grouped CTEs to avoid expensive full outer joins.

  • Groups by FileName (one CTE per raw table)
  • Aggregates per entity id column (URN/LA/England)
  • Coalesces properties only across the sources that actually contain them
  • Drops and recreates materialized views on regeneration
Key SQL pattern:
WITH src_1 AS (... GROUP BY "Id"),
     src_2 AS (... GROUP BY "Id")
SELECT
  COALESCE(src_1."Id", src_2."Id") AS "Id",
  COALESCE(src_1."SomeProp", src_2."SomeProp") AS "SomeProp"
FROM src_1
LEFT JOIN src_2 ON src_2."Id" = src_1."Id";

Operational workflow

1) Obtain source CSV extracts
2) Run cleaning step ‚Üí produces *.clean.csv (or equivalent)
3) Run GenerateRawTables.cs ‚Üí creates & loads raw_* tables
4) Run GenerateViews.cs ‚Üí produces materialized views
5) Refresh materialized views after load

Performance notes

  • Raw tables should have an index on the entity id column (URN/LA code)
  • If filters are frequently used, composite indexes including filter columns can help
  • Materialized views trade storage for query speed; refresh time is the main cost
